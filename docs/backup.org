* Overview

Nightly backup pipeline that runs on every NixOS server at 2:00 AM via a systemd timer. The pipeline orchestrates PostgreSQL dumps, ZFS snapshots, restic backup to S3, and reports on itself via Prometheus.

#+begin_example
  02:00 AM
     |
     v
  PostgreSQL dumps (pg_dump per database)   <-- services LIVE
     |
     v
  Stop services (serviceUnits)   ~~~~~~~~~~~~~~ stop window opens
     |
     v
  ZFS snapshot + mount (one call per pool, atomic)
     |
     v
  Restart services               ~~~~~~~~~~~~~~ stop window closes (~seconds)
     |
     v
  Restic backup to S3 (from mounted snapshots + extraPaths)
     |
     v
  Cleanup ZFS snapshots + mounts
     |
     v
  Write backup.prom for the textfile collector
     |
     v
  Done
#+end_example

Two design decisions carry most of the weight here.

*The stop window covers only the snapshot.* PostgreSQL dumps run first, with services live: =pg_dump= is MVCC-consistent by construction, so quiescing around it buys nothing, and it is by far the slowest step — on heartbeat the dumps take over ten minutes, of which =mailarchiver= alone is 8m14s for a 2.9G gzip. Previously the dumps ran inside the outage, so heartbeat was down 10m07s nightly. Now only the snapshot is, which is about a second.

*Services resume before the upload.* Restic reads immutable snapshot mounts, so the (slow) upload runs with everything back online.

Note the consequence for PostgreSQL: the dumps are captured by a snapshot taken ~10 minutes after they were written, so they do not line up with the blob datasets of the services whose rows they contain. That is deliberate, and safe, because the live data directory is *also* backed up — see [[*PostgreSQL: two artifacts]].

That last change also removed a cross-host outage. headscale sets =only_start_if_oidc_is_available= and exits fatally when its OIDC issuer is unreachable at startup; because every host fired its backup at 02:00:00, meshcontrol restarted headscale in the same minute accesscontrolsystem stopped authentik. On 2026-07-22 that crash-looped headscale eight times and dropped all 13 tailnet nodes for ~50s.

=commrelay= and =caretaker= have the pipeline disabled outright: neither has any dataset opting into restic, so both spent months backing up nothing while logging success.

* Architecture

** Module Structure

#+begin_example
modules/services/backup/
  options.nix   -- homelab.services.backup.* and homelab.services.restic.* options
  default.nix   -- orchestrator: systemd service + timer, pipeline script
  restic.nix    -- services.restic.backups.backup configuration (NixOS upstream module)
  zfs.nix       -- ZFS snapshot prepare/cleanup systemd services
#+end_example

** Systemd Units

| Unit                              | Type    | Purpose                                    |
|-----------------------------------+---------+--------------------------------------------|
| =backup-system.service=             | oneshot | Main orchestrator (runs the full pipeline) |
| =backup-system.timer=               | timer   | Fires at =*-*-* 02:00:00= daily                |
| =backup-restic-zfs-dataset-prepare= | oneshot | Snapshot ZFS datasets + mount read-only    |
| =backup-restic-zfs-dataset-cleanup= | oneshot | Unmount + destroy snapshots                |
| =restic-backups-backup.service=     | oneshot | Restic upload (NixOS-generated)            |
| =postgresqlBackup-<db>.service=     | oneshot | Per-database pg_dump (NixOS-generated)     |
| =backup-restic-check.service=       | oneshot | Repository integrity check                 |
| =backup-restic-check.timer=         | timer   | Fires =Sun *-*-* 04:00:00= weekly            |

The main =backup-system.service= invokes the other units via =systemctl start= rather than systemd dependencies. This gives the orchestrator script full control over ordering and error handling.

|--------------+-------------+---------------------+----------------------------------------------|

| =extraPaths=   | list of str | =[]          =      | Additional paths to include in restic backup |

*** =homelab.services.restic=

| =pruneOpts=              | list of str | =--keep-daily 3=, =--keep-weekly 4=       | Retention policy                       |
| =check.enable=           | bool        | =true=                                  | Run the integrity check on a timer     |
| =check.readDataSubset=   | str         | =10%=                                   | Fraction of pack data to download      |
| =check.onCalendar=       | str         | =Sun *-*-* 04:00:00=                    | When to check                          |

** ZFS Dataset Integration

Any ZFS dataset declared via =homelab.zfs.datasets.<name>= can opt into restic backup by setting =restic.enable = true=. The backup module automatically:

1. Discovers all datasets with =restic.enable = true=
2. Creates a snapshot named =<dataset>@restic_nightly=
3. Mounts the snapshot read-only at =<backupRoot>/<name>=
4. Includes the mount path in restic's backup paths
5. Cleans up (unmount + destroy snapshot) after backup completes

* Pipeline Details

** Service Stop/Start

The orchestrator reads service unit names from a Nix-generated text file. For each unit it:

1. Checks =LoadState= -- skips units that aren't installed (safe for shared configs)
2. Checks =is-active= -- skips units that are already stopped
3. Stops/starts the unit

An EXIT trap guarantees services are restarted even if the script fails at any point. A guard variable (=_services_started=) prevents double-restart when the explicit restart has already run.

** PostgreSQL Dumps

If =services.postgresql= and =services.postgresqlBackup= are enabled, the pipeline triggers each =postgresqlBackup-<db>.service= unit. These are standard NixOS =pg_dump= oneshots that write SQL dumps to the configured backup directory.

The dump directory is typically on its own ZFS dataset (with =restic.enable=), so the dumps are captured in the subsequent ZFS snapshot and uploaded by restic.

Individual database backup failures are logged but do not abort the pipeline.

** ZFS Snapshots

The prepare script runs in three phases, not a per-dataset loop. =zfs snapshot= is atomic across every name given in a single invocation, so all snapshots are taken together:

#+begin_src bash
# Phase 1 -- clear stale state from a previous failed run
umount /mnt/nightly_backup/<name> 2>/dev/null || true
zfs destroy <dataset>@restic_nightly 2>/dev/null || true

# Phase 2 -- snapshot, one call per pool, issued back to back
zfs snapshot <ds1>@restic_nightly <ds2>@restic_nightly <ds3>@restic_nightly

# Phase 3 -- mount read-only for restic
mount -t zfs -o ro <dataset>@restic_nightly /mnt/nightly_backup/<name>
#+end_src

This matters for services that own more than one dataset. =opencloud= spans five, and =matrix-synapse= splits its database and media store across two. Under the old loop those were captured up to a second apart, so metadata could reference storage that was not in its own snapshot — the loop only worked because services were stopped for the whole window, which they no longer are.

Atomicity is *per pool*: zfs rejects an invocation naming datasets in different pools. On heartbeat both multi-dataset services straddle =fpool= and =dpool=, so a residual window remains between the two calls — now the gap between two syscalls rather than a second of mounting. trialunit is single-pool, so all 19 of its datasets are captured in one atomic call.

The cleanup script reverses this: unmount, rmdir, destroy snapshot. It uses =set +euo pipefail= so cleanup proceeds even if individual steps fail.

** PostgreSQL: two artifacts

Both the live data directory and the =pg_dump= output are backed up, and they do different jobs:

| Artifact                     | Consistent with blobs? | Portable across PG majors? |
|------------------------------+------------------------+----------------------------|
| datadir (=postgresql= dataset) | Yes                    | No                         |
| dumps (=postgresql-backup=)    | No                     | Yes                        |

The *physical* copy is snapshotted in the same atomic per-pool call as every other service dataset, so it lines up exactly with the blobs those services reference. Restoring it is equivalent to recovering from a power cut, which is what =pg_basebackup= and PITR already rely on: =pg_wal= lives inside =$PGDATA=, and =$PGDATA= is the dataset mountpoint, so the WAL is inside the same snapshot. This is the primary restore path.

The *logical* dumps run before the stop window with services live, so they are minutes older than the snapshot that captures them and do not agree with the blob datasets. That is fine — they exist for the one case a physical restore cannot serve, which is restoring onto a different PostgreSQL major version.

Cost of carrying both, measured on heartbeat: 4.39G of datadir against a 639GiB repository, so +0.7%. The dump directory is already larger at 5.94G because =postgresqlBackup= keeps a =.prev= copy of every database.

** Restic Backup

The NixOS =services.restic.backups.backup= configuration:

- *Repository*: S3-compatible storage (credentials from sops)
- *Paths*: Mounted ZFS snapshots + any =extraPaths=
- *Connection tuning*: =s3.connections=10= for parallel uploads
- *Auto-initialize*: Creates the repo on first run
- *Timer disabled*: =timerConfig = null= (our orchestrator triggers it)
- *Retention*: 3 daily + 4 weekly snapshots (configurable via =pruneOpts=)

Note that =extraPaths= are read *live*, not from a snapshot — the pipeline restarts services before the upload runs. Keep them small and write-once. Restic also *silently skips paths that do not exist* rather than failing, which is how the authentik =extraPath= backed up nothing for five months; see [[*Monitoring]].

*** The path set matters more than it looks

Restic keys on the set of paths in a snapshot, and that has two separate consequences. Both bite when you add a dataset or edit =extraPaths=.

**Retention** is grouped by host *and path set*. A changed path set starts a new group, and the old group's final snapshots are pinned forever, because no new snapshots arrive to age them out under =keep-daily= / =keep-weekly=. Drop them by hand with =restic-backup forget <id>= if they accumulate.

**Parent selection** works the same way, and this one is expensive. =restic backup= picks its parent by matching host *and* paths; with no match it logs =no parent snapshot found, will read all files= and re-hashes everything rather than doing an incremental.

Measured on heartbeat when the PostgreSQL datadir was added to the path set:

| | |
|-------------------+------------|
| Data re-read      | 589.7 GB   |
| Repository before | 639.125 GiB |
| Repository after  | 642.477 GiB |
| Actual growth     | 3.35 GiB   |
| Wall clock        | 49 minutes (against ~1 minute steady state) |

It re-*reads*, it does not re-*upload* — content-defined chunking finds nearly every blob already present, so the growth was just the new datadir after dedup and compression. The cost is local I/O and CPU (1h32m of hashing), not egress.

So this is a one-time penalty, paid once per host, the first time a path set changes. The next run finds the new snapshot as its parent and goes incremental again. It is harmless — services are up throughout, since the upload runs after they restart, and =TimeoutStartSec= is 6h — but schedule it when you can leave the host alone for an hour rather than immediately before you need it.

** Integrity Check

=backup-restic-check.service= runs =restic check --read-data-subset= on its own weekly timer, decoupled from the nightly pipeline. It used to run inline at the end of every backup, which downloaded pack data from S3 on every host every night (3m31s on heartbeat) while still holding the snapshot mounts open — and a nightly 5% sample re-checks the same small slice far more often than a weekly larger one, while covering less of the repository over a month.

As a real unit, a failed check now marks the unit failed instead of being swallowed by =|| echo "(non-fatal)"=.

* Monitoring

The pipeline writes =backup.prom= into the textfile collector directory (=homelab.services.alloy.textfileDir=, default =/var/lib/alloy-textfile=) on every run, success or failure:

| Metric                                          | Meaning                                        |
|-------------------------------------------------+------------------------------------------------|
| =homelab_backup_success=                          | 1/0 for the most recent run                    |
| =homelab_backup_last_run_timestamp_seconds=       | When it last finished                          |
| =homelab_backup_last_success_timestamp_seconds=   | When it last actually worked                   |
| =homelab_backup_duration_seconds=                 | Wall clock                                     |
| =homelab_backup_paths_configured=                 | Paths restic was told to read                  |
| =homelab_backup_paths_present=                    | Paths that existed on disk at upload time      |

Alert on two things:

1. =homelab_backup_paths_present < homelab_backup_paths_configured= — something is being skipped silently.
2. =homelab_backup_last_success_timestamp_seconds= going stale.

Staleness matters more than failure. Every silent failure this pipeline has had was a *success* that backed up nothing: commrelay and caretaker running an empty pipeline for months, and the authentik =extraPath= pointing at a directory restic then skipped. A failure alert would have caught none of them. A failed run deliberately preserves the previous =last_success= value rather than overwriting it.

* Secrets

Each machine that runs backups has three restic secrets in its =secrets.sops.yaml=:

| Secret Path     | Purpose                                                         |
|-----------------+-----------------------------------------------------------------|
| =restic/.env=     | S3 credentials (=AWS_ACCESS_KEY_ID=, =AWS_SECRET_ACCESS_KEY=, etc.) |
| =restic/repo=     | Repository URL (e.g., =s3:https://garage.example.com/bucket=)     |
| =restic/password= | Restic encryption password                                      |

These are decrypted at runtime by sops-nix and passed to the restic service via =environmentFile=, =repositoryFile=, and =passwordFile=.

Machines with restic secrets: heartbeat, trialunit, accesscontrolsystem, meshcontrol, watchfulsystem. commrelay's declarations were dropped when its pipeline was disabled; the values remain in its =secrets.sops.yaml=.

* Safety Guarantees

1. *Services always restart*: EXIT trap fires on any failure; guard prevents double-start on success path
2. *Crash-consistent snapshots*: where services are stopped, they are stopped across the snapshot; where they are not, all of a service's state is in one dataset and the snapshot is atomic, which is the same guarantee the process must survive on power loss
3. *Services resume before upload*: The (slow) restic upload reads from immutable snapshot mounts, so services are back online during the upload window
4. *Stale state cleanup*: Prepare script cleans up leftover mounts/snapshots from previous failed runs before creating new ones
5. *Graceful degradation*: PostgreSQL backup failures and missing units are logged but do not abort the pipeline
6. *Always reports*: the EXIT trap writes metrics even on failure, so a run that dies partway is recorded as failed rather than leaving yesterday's success in place looking healthy

sudo systemctl start backup-system.service
#+end_src

#+begin_src bash
# Timer status
systemctl status backup-system.timer

# Restic repository stats

#+begin_src bash

#+end_src

** Run the Integrity Check On Demand

#+begin_src bash
sudo systemctl start backup-restic-check.service
journalctl -u backup-restic-check.service -f
#+end_src

** Manual Restic Operations

The NixOS restic module generates a =restic-backup= wrapper at =/run/current-system/sw/bin/restic-backup= with credentials and repository pre-configured.

#+begin_src bash
# List snapshots
sudo restic-backup snapshots

# Browse a snapshot
sudo restic-backup ls latest

# Restore a file
sudo restic-backup restore latest --target /tmp/restore --include /path/to/file

# Full integrity check
sudo restic-backup check --read-data

# Repository stats
sudo restic-backup stats
#+end_src

** Adding a New Machine

Before anything else: if the host has no dataset opting into restic and no PostgreSQL, it has nothing to back up. Leave =backup.enable = false= rather than running a pipeline that reports success for doing nothing.

1. Enable the backup module in =machines/<name>/_config.nix=:
   #+begin_src nix
   homelab = {
     services = {
       backup = {
         enable = true;
         # Default to []. A service only needs stopping if a snapshot of its
         # dataset would not be recoverable — i.e. it holds state in memory
         # that is never fsynced. Anything crash-safe (PostgreSQL, SQLite,
         # most modern daemons) does not belong here, and anything whose
         # state spans several datasets in different pools cannot be made
         # consistent by stopping it anyway.
         serviceUnits = [];
         # extraPaths = ["/var/lib/something"];  # read live, not snapshotted
       };
     };
   };
   #+end_src

2. Add restic secrets to =machines/<name>/secrets.sops.yaml=:
   #+begin_example
   restic:
       .env: ENC[AES256_GCM,...]
       repo: ENC[AES256_GCM,...]
       password: ENC[AES256_GCM,...]
   #+end_example

3. For ZFS-backed services, set =restic.enable = true= on the dataset:
   #+begin_src nix
   homelab = {
     services = {
       postgresql = {
         enable = true;
         backup = {
           enable = true;
           zfs = {
             enable = true;
             dataset = "rpool/root/var/backup/postgresql";
           };
         };
       };
     };
   };
   #+end_src
