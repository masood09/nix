* Overview

Three NixOS desktops run a Wayland-native stack: Niri compositor, greetd login manager, PipeWire audio, and Stylix theming. commandmodule and sonic use ZFS root with impermanence (ephemeral root via ZFS rollback). arrakis uses LUKS-encrypted ext4 with tmpfs impermanence and systemd-boot. macOS desktops (murderbot, work-pantheon) use nix-darwin with Stylix for HM-level theming (fonts, opacity, terminal colors) but are otherwise out of scope for this document.

| Machine       | Hardware           | GPU            | Primary User       | Theme               | Root FS    | Boot                    | Fingerprint |
|---------------+--------------------+----------------+--------------------+---------------------+------------+-------------------------+-------------|
| arrakis       | ThinkPad T14 Gen 3 | AMD Radeon     | masoodahmed (1000) | gruvbox-dark        | ext4/LUKS  | systemd-boot + Plymouth | no          |
| commandmodule | ThinkPad (Intel)   | Intel UHD 620  | masoodahmed (1000) | gruvbox-dark        | ZFS        | GRUB                    | yes         |
| sonic         | ThinkPad T14 Gen 3 | AMD Ryzen iGPU | zainahmed (1001)   | sonic-dark (custom) | ZFS        | GRUB                    | yes         |

* Module Structure

Desktop configuration is split across NixOS system modules and home-manager user modules.

#+begin_example
modules/nixos/
  _stylix.nix        Stylix — single source of truth for Base16 theming (system + HM via autoImport)

modules/nixos/desktop/
  default.nix        Shared services, hardware, kernel (Zen), VM tuning, ananicy-cpp, fontconfig, fonts, Bitwarden polkit action; defines desktop.shell option
  _greetd.nix        Login manager (tuigreet + auto-login for encrypted non-ZFS desktops) with PAM config
  _niri.nix          Niri compositor + backlight control (light)
  _noctalia.nix      Noctalia desktop shell (system package, guarded on desktop.shell)

modules/macos/
  _stylix.nix        Stylix — Darwin counterpart (same options, no cursor/Plymouth/GRUB/console)

modules/home-manager/
  home.nix           Base HM config: home path, session vars, XDG user directories, fontconfig Nerd Font override

modules/home-manager/programs/
  niri/default.nix   Niri HM config (programs.niri.settings, rofi, swaync, swaylock, playerctld, udiskie, user services; inert on Darwin)
  niri/_noctalia.nix Noctalia HM module enablement and settings (guarded on desktop.shell == "noctalia")
  niri/_waybar.nix   Waybar status bar (workspaces, window title, tray, audio, network, battery, clock, notifications)
  _kitty.nix         Terminal emulator
  zen/               Browser (privacy hardening, extensions, containers, pinned tabs)

machines/commandmodule/
  _niri.nix          Machine-local Niri laptop panel output placement
machines/sonic/
  _niri.nix          Machine-local Niri laptop panel output placement
#+end_example

Everything is gated behind ~homelab.*~ options. A machine opts into the desktop stack by setting ~homelab.desktop.enable = true~ and ~homelab.desktop.niri.enable = true~ in its ~_config.nix~. ~homelab.desktop.shell~ (enum, default ~"noctalia"~) selects a desktop shell; when set, shell-provided desktop pieces such as the bar, notifications, lock screen UI, wallpaper, and other replacement surfaces are skipped. ~swayidle~ still remains in the session for pre-suspend locking, and ~rofi~ remains enabled so the launcher stays available on ~Mod+D~ regardless of shell choice.

* Login Flow

** Greeter

greetd manages login sessions for all NixOS desktops. By default it runs tuigreet, a TUI greeter that launches ~niri-session~. On encrypted non-ZFS desktops (~homelab.isEncryptedRoot = true~ and ~homelab.isRootZFS = false~), greetd instead starts ~initial_session~ for the primary user immediately after boot-time disk unlock. Supports fingerprint auth when ~homelab.hardware.fingerprint.enable = true~.

Today that means fingerprint auth is available on ~commandmodule~ and ~sonic~. ~arrakis~ is explicitly password-only.

** PAM Stack

The greetd PAM service is configured in ~_greetd.nix~:

- ~enableGnomeKeyring = true~ — passes the login password to gnome-keyring-daemon for auto-unlock
- ~fprintAuth~ — conditional on hardware fingerprint support
- ~pam_fde_boot_pw.so inject_for=gkr~ (session, optional) — added only on encrypted non-ZFS desktops (~autoLoginFromBootPassword = true~). Ordered to run just before ~pam_gnome_keyring~'s session rule. Sourced from ~nixpkgs-unstable~ because the package has not landed in ~nixos-25.11~ yet.

PAM evaluation order for tuigreet with fingerprint enabled:

1. ~pam_fprintd.so~ (sufficient) — if fingerprint succeeds, PAM short-circuits
2. ~pam_unix.so~ (optional, early) — password collection
3. ~pam_gnome_keyring.so~ (optional) — receives password for keyring unlock
4. ~pam_unix.so~ (sufficient, try_first_pass) — password verification

** GNOME Keyring

~services.gnome.gnome-keyring~ is enabled for all desktops. The PAM module auto-unlocks the ~login~ keyring on password login. Fingerprint login cannot unlock the keyring because no password flows through PAM (see Pain Points).

On encrypted non-ZFS desktops (currently just ~arrakis~), auto-login from the boot password is covered by ~pam_fde_boot_pw.so inject_for=gkr~ in the greetd session stack. ~systemd-cryptsetup~ runs inside the systemd initrd (~boot.initrd.systemd.enable = true~ via the ~isDesktopSystemdBoot~ guard in ~modules/nixos/_boot.nix~) and caches the LUKS passphrase in the kernel keyring under the ~cryptsetup~ description. ~pam_fde_boot_pw~ reads that cached passphrase at session start and injects it so ~pam_gnome_keyring~ can unlock the ~login~ keyring as if the user had typed it. This requires the user's login password to match the LUKS passphrase; if they diverge, the session starts but the keyring stays locked (apps fall back to prompting).

** Desktop Applications

Bitwarden Desktop is installed from ~modules/home-manager/programs/_packages.nix~ as part of the shared Linux desktop package set. Element Desktop is managed by Home Manager's native ~programs.element-desktop~ module in ~modules/home-manager/programs/_element-desktop.nix~ and is enabled per machine through ~homelab.programs.element-desktop.enable~. The module is designed to be reusable on both Linux and macOS, but on ~murderbot~ it is currently disabled because nixpkgs ~element-desktop~ fails to build on Darwin (see "Element on Darwin" below); Element is provided there by the Homebrew ~element~ cask as a temporary workaround.

Bitwarden also needs machine-wide integration: fingerprint-capable NixOS desktops install a generated polkit action (~com.bitwarden.Bitwarden.unlock~) from ~modules/nixos/desktop/default.nix~ so Bitwarden's "system authentication" unlock path can delegate to polkit instead of failing because the action is undefined. In the current fleet that action is present on ~commandmodule~ and ~sonic~, but not on ~arrakis~.

Element Desktop currently has no comparable system-wide integration in this repo, but on Linux its Home Manager module overrides the package to start Electron with ~--password-store=gnome-libsecret~. That forces Electron onto GNOME Keyring's Secret Service backend in the Niri session, where Electron's desktop auto-detection would otherwise fail and offer weaker local encryption. The module also declares a fuller ~settings~ baseline: it points Element at ~homelab.programs.element-desktop.baseUrl~ and ~homelab.programs.element-desktop.serverName~ (defaulting to ~https://chat.mantannest.com~ / ~chat.mantannest.com~), locks login to that homeserver, defaults to the dark theme, and leaves analytics disabled by not configuring ~posthog~. Using the native HM module also leaves room for future declarative extra ~profiles~ without pushing that app-specific config back into the generic package list.

*** Element on Darwin (tech debt)

On macOS the HM ~programs.element-desktop~ module is intentionally not enabled. nixpkgs ~element-desktop@1.12.10~ fails to build on Darwin: ~electron-builder@26.x~ added an eager actool version check in ~app-builder-lib/src/util/macosIconComposer.ts~ that runs at module load time and calls bare ~actool~ with no absolute path. The Nix builder's PATH is built only from ~nativeBuildInputs~, so ~/usr/bin/actool~ is unreachable from inside the build even with ~sandbox = false~ — the build fails with "Failed to check actool version. Is Xcode 26 or higher installed?" regardless of whether Xcode is installed on the host. Because ~platformPackager.ts~ imports ~macosIconComposer~ unconditionally, the failure happens even though Element never goes near the macOS icon composer.

As a temporary workaround ~murderbot~ installs Element via the Homebrew ~element~ cask (see ~machines/murderbot/_packages.nix~) and disables the HM module entry in ~machines/murderbot/default.nix~. The long-term direction is to drop the cask and re-enable the HM module once nixpkgs picks up an upstream electron-builder fix or patches the bare ~actool~ call to use an absolute path. The HM-managed ~settings~ baseline does not apply on Darwin while the module is disabled; if homeserver pinning becomes important on macOS before the upstream fix lands, the cleanest fix is a downstream overlay on ~pkgs.element-desktop~ rather than re-enabling the broken build.

* Compositor

Niri is a scrollable-tiling Wayland compositor. System-level enablement (~programs.niri.enable~ and ~programs.light.enable~ for backlight control) is in ~_niri.nix~. The package/session integration comes from ~sodiboo/niri-flake~'s NixOS module, included via ~mkNixOSDesktopConfig~ rather than in the shared ~mkNixOSConfig~ builder. This is intentional: the niri-flake NixOS module unconditionally injects its HM module into ~home-manager.sharedModules~ whenever home-manager is present, which would force the niri package into every server closure even with ~programs.niri.enable = false~. User configuration is managed declaratively through home-manager via ~programs.niri.settings~, and Stylix themes Niri through the same flake integration.

When Niri is enabled on Linux, home-manager configures: Niri settings, rofi (app launcher), waybar (status bar managed by systemd, configured in ~_waybar.nix~ with custom CSS using Stylix base16 color variables and ~addCss = false~ to avoid layout conflicts), swaync (notifications), swaylock (screen locker), playerctld (MPRIS daemon), and udiskie (auto-mount). Packages without HM modules (wl-clipboard, xwayland-satellite, swaybg, swayidle) are installed directly. ~xwayland-satellite~ is always started as a user service. ~swaybg~ is started as a user service only when ~desktop.shell = "none"~; ~swayidle~ remains installed in all Niri sessions because both shell and non-shell modes use it for session-side before-sleep locking.

The Niri HM module is still imported for macOS users because the shared HM tree is imported uniformly across Linux and Darwin machines. The module therefore cannot rely on conditional imports to stay Darwin-safe. Instead it checks whether ~options.programs.niri~ exists and only materializes the ~programs.niri~ subtree when Home Manager's upstream Niri module is actually present. This avoids nix-darwin evaluation failures from defining an option path that does not exist on Darwin.

~homelab.desktop.shell~ (enum, default ~"noctalia"~) selects a desktop shell that replaces individual bar/notification/lock/wallpaper programs. When set to anything other than ~"none"~, waybar, swaync, swaylock, swaybg, and udiskie are skipped. ~swayidle~ remains available for session-side before-sleep locking, ~rofi~ remains enabled, and ~Mod+D~ continues to launch it regardless of shell selection. Compositor-level utilities (wl-clipboard, xwayland-satellite), ~swayidle~, and playerctld remain unconditional.

When ~desktop.shell = "noctalia"~, the shared Niri keymap routes shell-owned hardware controls through ~noctalia-shell ipc call ...~ instead of direct helper binaries: lock, volume, mute, media play/pause/previous/next, brightness, Wi-Fi, Bluetooth, settings, and power-profile cycling all go through Noctalia so the shell owns the resulting state and OSD. The only remaining direct media fallback is ~XF86AudioStop~, because the current Noctalia IPC surface does not expose a native stop action. Outside Noctalia, the same Niri config falls back to ~swaylock~, ~wpctl~, ~playerctl~, and ~light~ as appropriate. ~XF86Favorites~ mirrors ~Mod+D~ globally and still launches ~rofi -show drun~ regardless of shell.

When Noctalia is selected, Niri also starts the shell explicitly via ~programs.niri.settings.spawn-at-startup = [{ command = [ "noctalia-shell" ]; }]~. This keeps shell startup tied to the compositor session, which matches Noctalia's current Niri guidance and avoids enabling the shell's experimental systemd user service in parallel. A separate ~noctalia-before-sleep-lock~ user service runs ~swayidle -w before-sleep ...~ with Noctalia IPC so lid-close suspend locks while the user session is still alive; earlier root-side sleep hooks and generic ~loginctl lock-session~ wiring were intentionally removed because they ran too late or did not integrate reliably with the active Noctalia session.

The shared Niri config defines the external LG 4K monitor using a stable logical key and matches the actual hardware by monitor name (~LG Electronics LG ULTRAFINE 110NTHM20241~) rather than a transient connector like ~DP-2~. Laptop panel placement remains machine-local in ~machines/{commandmodule,sonic}/_niri.nix~ so the shared desktop module only owns output state that is common across both laptops.

The primary directional bindings use edge-aware fallbacks so movement stays continuous across the desktop topology. ~Mod+Left/Right~ and ~Mod+H/L~ focus the adjacent column, then fall through to the monitor on that side when already at the edge. ~Mod+Up/Down~ and ~Mod+K/J~ focus the adjacent window, then fall through to the workspace above or below when already at the first or last window. The matching ~Mod+Shift+...~ bindings move the focused column or window with the same fallback behavior, while the ~Mod+Ctrl+...~ bindings remain explicit monitor navigation.

* Noctalia Shell

[[https://noctalia.dev/][Noctalia]] is a minimal Wayland desktop shell built on Quickshell. It provides panels, notifications, and system UI with a clean aesthetic and 83+ community plugins. Noctalia supports Niri natively, making it a natural fit for the current compositor stack. Enabled by default (~homelab.desktop.shell = "noctalia"~). The system package is installed via ~modules/nixos/desktop/_noctalia.nix~; the HM module is included via ~mkNixOSDesktopConfig~'s ~home-manager.sharedModules~ and activated in ~modules/home-manager/programs/niri/_noctalia.nix~; and Niri starts the shell via ~spawn-at-startup~ in ~modules/home-manager/programs/niri/default.nix~. The repo intentionally does not enable Noctalia's experimental systemd user service so the shell has a single startup path. Requires nixos-unstable (provided via the ~nixpkgs-unstable~ flake input) and uses the ~noctalia.cachix.org~ binary cache. The plugin registry (~plugins.json~) is managed declaratively via ~xdg.configFile~; the ~model-usage~ plugin is driven by ~homelab.programs.ai_tools.models~, but providers that read local CLI state still require the matching tool to be installed. In particular, the OpenAI/Codex widget path only turns on when the machine selects the ~codex~ or ~openai~ model and also installs the ~codex~ CLI, because the upstream plugin reads usage from =~/.codex=. The companion ~homelab.programs.ai_tools.tools~ enum list controls which assistant CLIs are installed. The default bar layout now follows the GUI-customized arrangement captured from the running shell: the tray lives on the right with its drawer disabled and icons left uncolourized, the bar stays in floating mode, the volume widget is always shown, and the Bluetooth widget is only rendered on machines where ~homelab.hardware.bluetooth.enable = true~. The generated ~settings.json~ also pins Noctalia's current ~settingsVersion~ so fresh launches do not re-run legacy migrations and silently coerce declarative bar settings back to upstream defaults.

* Input

Keyboard repeat is tuned to match macOS so the typing feel is consistent across the laptop fleet (NixOS desktops + Darwin machines). macOS is the source of truth — Darwin's defaults app uses 15ms ticks for ~InitialKeyRepeat~ and ~KeyRepeat~, and the equivalent libinput values are mirrored on Linux desktops via Niri's ~input.keyboard.repeat-delay~ / ~repeat-rate~.

| Setting       | Darwin (modules/macos/base.nix) | Linux (niri input.keyboard) | Effective value         |
|---------------+---------------------------------+-----------------------------+-------------------------|
| Initial delay | ~InitialKeyRepeat = 15~ (ticks)   | ~repeat-delay = 225~ (ms)     | 225 ms                  |
| Repeat rate   | ~KeyRepeat = 2~ (ticks)           | ~repeat-rate = 33~ (Hz)       | ~30 ms / repeat (~33Hz) |

Conversion: Darwin ticks × 15ms = Linux ms; ~1000 / (KeyRepeat * 15)~ ≈ Linux Hz. If you change one side, change the other — both files carry inline comments pointing at each other.

Niri's defaults (600ms delay, 25Hz) feel sluggish next to macOS, which is why we override here rather than relying on upstream defaults. Greetd's TTY prompt is unaffected, but the desktop session begins immediately after login so it does not matter in practice.

* Graphics

GPU acceleration is configured per-driver via ~homelab.hardware.graphics.driver~:

** Intel (commandmodule)

| Package               | Purpose                                 |
|-----------------------+-----------------------------------------|
| intel-media-driver    | iHD VA-API (hardware video, Broadwell+) |
| intel-vaapi-driver    | i965 VA-API fallback (older Intel)      |
| intel-compute-runtime | OpenCL (NEO)                            |
| libva-vdpau-driver    | VDPAU compatibility over VA-API         |
| libvdpau-va-gl        | VDPAU fallback via OpenGL               |

~LIBVA_DRIVER_NAME=iHD~. Kernel param ~i915.enable_psr=0~ disables Panel Self Refresh to fix scrolling micro-stutter on Whiskey Lake.

** AMD (sonic)

| Package        | Purpose                   |
|----------------+---------------------------|
| libvdpau-va-gl | VDPAU fallback via OpenGL |

~LIBVA_DRIVER_NAME=radeonsi~. Mesa's built-in RADV/radeonsi handles VA-API natively — no separate VA-API or OpenCL driver is needed for desktop use. ~hardware.amdgpu.initrd.enable = true~ for early module loading. Kernel param ~amd_pstate=active~ forces the modern AMD P-State EPP driver for fine-grained frequency scaling.

* Audio

PipeWire replaces PulseAudio and ALSA with a unified daemon. Enabled when ~homelab.hardware.audio.enable = true~:

- ALSA compatibility with 32-bit support (Wine/Steam)
- PulseAudio compatibility layer
- RealtimeKit (~security.rtkit~) for realtime scheduling priority

* Theming

Stylix provides Base16 theming from a single source of truth at the system level (~modules/nixos/_stylix.nix~ on NixOS, ~modules/macos/_stylix.nix~ on Darwin). All settings are defined once under ~homelab.stylix.*~ and propagate to Home-Manager automatically via Stylix's ~autoImport~ (injects ~homeModules.stylix~ for every HM user) and ~followSystem~ (copies values with ~mkDefault~ priority).

Stylix is enabled by default for all machines. Override ~homelab.stylix.enable = false~ only when a host needs to opt out.

| Setting    | Default                        | Notes                                |
|------------+--------------------------------+--------------------------------------|
| Scheme     | gruvbox-dark                   | String (named) or path (custom YAML) |
| Polarity   | dark                           | ~dark~ or ~light~                        |
| Monospace  | JetBrainsMono Nerd Font (12pt) | Nerd Font variant                    |
| Sans-serif | Inter (12pt)                   |                                      |
| Cursor     | Bibata Modern Classic          | Size 24 (NixOS only, not on Darwin)  |
| Opacity    | 0.90                           | Terminal and popups                  |

NixOS system-level targets (Plymouth, GRUB, console) auto-enable when their underlying program is active. HM-only target overrides (starship disabled, waybar ~addCss = false~, zen-browser profile) are injected via ~home-manager.sharedModules~ in the system-level module.

sonic uses a custom scheme (~nix/themes/sonic-dark.yaml~) with a Sonic the Hedgehog-inspired palette.

Starship prompt is excluded from Stylix targets — it uses hardcoded Catppuccin colors. Fish uses tide instead of Starship; tide prompt colors are derived from Stylix base16 slots in ~_fish.nix~ so they adapt automatically per scheme. The prompt layout (style, separators, icons, transient prompt) is set by a home-manager activation script that runs ~tide configure --auto~ after ~writeBoundary~. The ~--auto~ flags are version-coupled to the tide release in nixpkgs (currently v6.2.0) — each flag maps to a choice file under ~functions/tide/configure/choices/~ in the tide repo, and a nixpkgs bump that adds or renames a choice will cause the activation to fail visibly.

A fontconfig override in ~home.nix~ redirects common monospace fonts (Liberation Mono, DejaVu Sans Mono, Noto Sans Mono) to the Stylix monospace font to preserve Nerd Font PUA icon glyphs in web pages and apps.

* Persistence

All desktops use impermanence with ~/nix/persist~ as the persistence root. ZFS desktops (commandmodule, sonic) roll back root to a blank snapshot on every boot (~zfs rollback -r rpool/root/empty@start~). Non-ZFS desktops (arrakis) use tmpfs root, which is inherently wiped on reboot. Desktop-specific persistent directories:

| Path                                   | Purpose                                                        |
|----------------------------------------+----------------------------------------------------------------|
| /var/lib/bluetooth                     | Bluetooth pairings                                             |
| /var/lib/fprint                        | Fingerprint templates (used on fingerprint-enabled desktops)   |
| /var/lib/NetworkManager                | WiFi connections                                               |
| /etc/NetworkManager/system-connections | WiFi connection configs                                        |
| /var/cache/tuigreet                    | Greeter session cache                                          |
| /home/<user>                           | User home (non-ZFS: auto-created with ownership; ZFS: dataset) |

* Kernel & Performance

Desktops use the Zen kernel (~linuxPackages_zen~) for improved interactive responsiveness. Key differences from the stock kernel:

- BORE scheduler — lower latency for foreground applications
- 1000Hz tick rate — smoother input handling and animations
- Optimized preemption — less stalling under load

ZFS 2.3.x is compatible with the Zen kernel. If issues arise, select an older generation from GRUB to revert.

** VM Tuning (sysctl)

Desktop-specific sysctl overrides inspired by CachyOS defaults. Absolute byte dirty limits replace ratio-based settings for predictable write-back regardless of RAM size. zram-aware values suit compressed swap.

| Parameter                   |     Value | Default | Rationale                                           |
|-----------------------------+-----------+---------+-----------------------------------------------------|
| vm.swappiness               |       150 |      60 | Prefer compressing to zram over evicting file cache |
| vm.page-cluster             |         0 |       3 | Single-page swap reads suit zram (no seek penalty)  |
| vm.dirty_background_bytes   |  67108864 |       0 | Flush at 64 MB background (predictable write-back)  |
| vm.dirty_bytes              | 268435456 |       0 | Hard cap at 256 MB (predictable write-back)         |
| vm.vfs_cache_pressure       |        50 |     100 | Retain more filesystem metadata in cache            |
| vm.compaction_proactiveness |         0 |      20 | Avoid background compaction latency spikes          |
| vm.min_free_kbytes          |     67584 |    4096 | Larger free-memory reserve to prevent alloc stalls  |
| kernel.nmi_watchdog         |         0 |       1 | Saves power, reduces interrupts                     |
| kernel.split_lock_mitigate  |         0 |       1 | Avoids split-lock detection performance penalty     |

** Process Scheduling (ananicy-cpp)

ananicy-cpp is an auto-nice daemon that adjusts process niceness, I/O priority, and CPU scheduling class in real time. It uses the CachyOS community ruleset (~ananicy-rules-cachyos~) which covers hundreds of common applications. A custom rule is added for ~zen-beta~ (mapped to ~Doc-View~ type, same as Firefox/Zen) since upstream only covers ~zen~ and ~zen-bin~.

** Font Rendering (fontconfig)

| Setting   | Value   | Rationale                                                         |
|-----------+---------+-------------------------------------------------------------------|
| antialias | true    | Smooth font edges                                                 |
| hinting   | slight  | Respects built-in font hints (JetBrains Mono, etc.)               |
| autohint  | false   | Disabled — fonts with good hints render better without            |

* Boot

ZFS desktops use GRUB (~gfxmodeEfi = "auto"~ for graphical framebuffer); Stylix themes the GRUB menu with Base16 colors by default. Non-ZFS desktops (arrakis) use systemd-boot with systemd-based initrd and Plymouth. The boot sequence there is: systemd-boot → Plymouth splash (themed by Stylix) → LUKS passphrase (via Plymouth's password agent) → desktop session. If the machine explicitly disables Stylix, the boot path remains the same but the Base16 theming pieces fall away. If the machine is an unencrypted or ZFS desktop, greetd falls back to the normal greeter path instead. Kernel params ~quiet splash loglevel=3~ suppress console output so the splash is uninterrupted. Plymouth and systemd initrd are auto-enabled for any machine with ~role = "desktop"~ and ~isRootZFS = false~ (the ~isDesktopSystemdBoot~ guard in ~_boot.nix~).

* Pain Points

** Fingerprint login cannot unlock GNOME Keyring

When ~pam_fprintd.so~ succeeds (marked ~sufficient~), PAM short-circuits the auth stack. ~pam_gnome_keyring.so~ never receives a password, so the ~login~ keyring stays locked. Apps that use the Secret Service API (Element, Discord) will prompt for credentials each session.

Note that the analogous auto-login pain point on encrypted non-ZFS desktops (~arrakis~) is now resolved by ~pam_fde_boot_pw~ in the greetd session stack — see Login Flow / GNOME Keyring above. The same trick does not help fingerprint logins because ~commandmodule~ and ~sonic~ are ZFS desktops with no LUKS passphrase cached in the kernel keyring; the only password source available to PAM during a fingerprint login is the user's typed password, which is exactly what fingerprint replaces.

*Workaround:* log in with password instead of fingerprint. The keyring unlocks automatically.

** Bitwarden system-auth unlock is only wired on fingerprint desktops

The repo currently installs ~com.bitwarden.Bitwarden.unlock~ only when ~homelab.hardware.fingerprint.enable = true~. That matches the immediate requirement: make Bitwarden biometric unlock work on laptops with fingerprint readers. If a future non-fingerprint Linux desktop wants Bitwarden's polkit-backed system-auth flow via password only, this guard should be widened and the behavior re-evaluated.

** ZFS requires GRUB

systemd-boot does not support ZFS. commandmodule and sonic are locked to GRUB. arrakis has migrated to ext4+LUKS and uses systemd-boot.
