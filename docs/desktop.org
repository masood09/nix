* Overview

Two NixOS desktops run a Wayland-native stack: Niri compositor, greetd login manager, PipeWire audio, and Stylix theming. Both use LUKS-encrypted ext4 with tmpfs impermanence and systemd-boot. The sole macOS machine (work-okta) uses nix-darwin with Stylix for HM-level theming (fonts, opacity, terminal colors) but is otherwise out of scope for this document.

| Machine       | Hardware           | GPU            | Primary User       | Admin User         | Theme               | Root FS    | Boot                    | Fingerprint |
|---------------+--------------------+----------------+--------------------+--------------------+---------------------+------------+-------------------------+-------------|
| usul          | ThinkPad T14 Gen 3 | AMD Radeon     | masoodahmed (1000) | (same)             | gruvbox-dark        | ext4/LUKS  | systemd-boot + Plymouth | no          |
| sonic         | ThinkPad T490      | Intel UHD 620  | zainahmed (1001)   | masoodahmed (1000) | sonic-dark (custom) | ext4/LUKS  | systemd-boot + Plymouth | yes         |

* Module Structure

Desktop configuration is split across NixOS system modules and home-manager user modules.

#+begin_example

Everything is gated behind ~homelab.*~ options. A machine opts into the desktop stack by setting ~homelab.desktop.enable = true~ and ~homelab.desktop.niri.enable = true~ in its ~_config.nix~. ~homelab.desktop.gaming.enable = true~ adds Steam, Gamescope, GameMode, and the supporting driver/tool stack (see Gaming section below). ~homelab.desktop.shell~ (enum, default ~"noctalia"~) selects a desktop shell; when set, shell-provided desktop pieces such as the bar, notifications, lock screen UI, wallpaper, and other replacement surfaces are skipped. ~swayidle~ still remains in the session for pre-suspend locking, and ~rofi~ remains enabled so the launcher stays available on ~Mod+D~ regardless of shell choice.

* Login Flow

** Greeter

greetd manages login sessions for all NixOS desktops. By default it runs tuigreet, a TUI greeter that launches ~niri-session~. On encrypted non-ZFS desktops (~homelab.isEncryptedRoot = true~ and ~homelab.isRootZFS = false~), greetd instead starts ~initial_session~ for the primary user immediately after boot-time disk unlock. Supports fingerprint auth when ~homelab.hardware.fingerprint.enable = true~.

Today that means fingerprint auth is available on ~sonic~. ~usul~ is explicitly password-only.

** PAM Stack

The greetd PAM service is configured in ~_greetd.nix~:

- ~enableGnomeKeyring = true~ — passes the login password to gnome-keyring-daemon for auto-unlock
- ~fprintAuth~ — conditional on hardware fingerprint support
- ~pam_fde_boot_pw.so inject_for=gkr~ (session, optional) — added only on encrypted non-ZFS desktops (~autoLoginFromBootPassword = true~). Ordered to run just before ~pam_gnome_keyring~'s session rule.

PAM evaluation order for tuigreet with fingerprint enabled:

1. ~pam_fprintd.so~ (sufficient) — if fingerprint succeeds, PAM short-circuits
2. ~pam_unix.so~ (optional, early) — password collection
3. ~pam_gnome_keyring.so~ (optional) — receives password for keyring unlock
4. ~pam_unix.so~ (sufficient, try_first_pass) — password verification

** GNOME Keyring

~services.gnome.gnome-keyring~ is enabled for all desktops. The PAM module auto-unlocks the ~login~ keyring on password login. Fingerprint login cannot unlock the keyring because no password flows through PAM (see Pain Points).

On encrypted non-ZFS desktops (currently just ~usul~), auto-login from the boot password is covered by ~pam_fde_boot_pw.so inject_for=gkr~ in the greetd session stack. ~systemd-cryptsetup~ runs inside the systemd initrd (~boot.initrd.systemd.enable = true~ via the ~isDesktopSystemdBoot~ guard in ~modules/nixos/_boot.nix~) and caches the LUKS passphrase in the kernel keyring under the ~cryptsetup~ description. ~pam_fde_boot_pw~ reads that cached passphrase at session start and injects it so ~pam_gnome_keyring~ can unlock the ~login~ keyring as if the user had typed it. This requires the user's login password to match the LUKS passphrase; if they diverge, the session starts but the keyring stays locked (apps fall back to prompting).

** Desktop Applications

Bitwarden Desktop is installed from ~modules/home-manager/programs/_packages.nix~ as part of the shared Linux desktop package set. Element Desktop is managed by Home Manager's native ~programs.element-desktop~ module in ~modules/home-manager/programs/_element-desktop.nix~ and is enabled per machine through ~homelab.programs.element-desktop.enable~.

Bitwarden also needs machine-wide integration: NixOS desktops that set ~homelab.programs.bitwarden.systemAuthUnlock.enable = true~ install a generated polkit action (~com.bitwarden.Bitwarden.unlock~) from ~modules/nixos/desktop/default.nix~ so Bitwarden's "system authentication" unlock path can delegate to polkit instead of failing because the action is undefined. The companion ~passwordless~ option sets the action's ~allow_active~ to ~yes~ (silent auto-unlock at launch) instead of ~auth_self~ (polkit prompt); ~allow_any~/~allow_inactive~ stay at ~auth_self~ so remote/inactive sessions always authenticate. Silent unlock relies on the login keyring already being open at boot (~pam_fde_boot_pw~), and the tradeoff is that any process in the logged-in session can then trigger an unlock without authenticating — acceptable only on autologin desktops whose session is already trusted. In the current fleet both ~usul~ and ~sonic~ set ~enable = true~ and ~passwordless = true~. Enabling the option is orthogonal to ~hardware.fingerprint.enable~; on ~sonic~ ~passwordless~ deliberately bypasses the fingerprint prompt too.

Note the app-side prerequisite that Nix cannot manage: in Bitwarden's own settings you must enable "Unlock with system authentication" once per machine, since that state lives in the app's data directory rather than in this repo.

*Status (2026-07): the polkit half of this is correct and deployed, but end-to-end silent auto-unlock does not actually work yet on our stack — see "Bitwarden system-auth auto-unlock does not work end-to-end" under Pain Points. The ~systemAuthUnlock~ option is kept in place, dormant, for when the upstream blockers clear.*

Element Desktop currently has no comparable system-wide integration in this repo, but on Linux its Home Manager module overrides the package to start Electron with ~--password-store=gnome-libsecret~. That forces Electron onto GNOME Keyring's Secret Service backend in the Niri session, where Electron's desktop auto-detection would otherwise fail and offer weaker local encryption. The module also declares a fuller ~settings~ baseline: it points Element at ~homelab.programs.element-desktop.baseUrl~ and ~homelab.programs.element-desktop.serverName~ (defaulting to ~https://chat.mantannest.com~ / ~chat.mantannest.com~), locks login to that homeserver, defaults to the dark theme, and leaves analytics disabled by not configuring ~posthog~. Using the native HM module also leaves room for future declarative extra ~profiles~ without pushing that app-specific config back into the generic package list.

* Compositor

Niri is a scrollable-tiling Wayland compositor. System-level enablement (~programs.niri.enable~, ~programs.light.enable~ for backlight control, GTK portal backend, and FileChooser routing) is in ~_niri.nix~. The package/session integration comes from ~epireyn/niri-flake~'s NixOS module (a fork of ~sodiboo/niri-flake~ that stays current and publishes ~niri-epireyn.cachix.org~, which covers nixos-26.05 — the input therefore carries no nixpkgs ~follows~), included via ~mkNixOSDesktopConfig~ rather than in the shared ~mkNixOSConfig~ builder. This is intentional: the niri-flake NixOS module unconditionally injects its HM module into ~home-manager.sharedModules~ whenever home-manager is present, which would force the niri package into every server closure even with ~programs.niri.enable = false~. User configuration is managed declaratively through home-manager via ~programs.niri.settings~, and Stylix themes Niri through the same flake integration.

~_niri.nix~ also adds ~xdg-desktop-portal-gtk~ to ~xdg.portal.extraPortals~ and explicitly routes ~org.freedesktop.impl.portal.FileChooser~ to the ~gtk~ backend via ~xdg.portal.config.niri~. niri-flake's portal config (~niri-portals.conf~) sets ~default=gnome;gtk;~, so the GNOME backend gets first pick for all interfaces. The GNOME backend does implement FileChooser, but its dialog requires a full GNOME session to render — on a bare niri compositor it accepts the D-Bus request then silently fails without falling back to GTK. The explicit routing override ensures "Save As" dialogs (e.g. browser downloads configured with ~browser.download.useDownloadDir = false~) always appear. macOS is unaffected because its native file picker does not go through xdg-desktop-portal.

When Niri is enabled on Linux, home-manager configures: Niri settings, rofi (app launcher), waybar (status bar managed by systemd, configured in ~_waybar.nix~ with custom CSS using Stylix base16 color variables and ~addCss = false~ to avoid layout conflicts), swaync (notifications), swaylock (screen locker), playerctld (MPRIS daemon), and udiskie (auto-mount). Packages without HM modules (wl-clipboard, xwayland-satellite, swaybg, swayidle) are installed directly. ~xwayland-satellite~ is always started as a user service. ~swaybg~ is started as a user service only when ~desktop.shell = "none"~; ~swayidle~ remains installed in all Niri sessions because both shell and non-shell modes use it for session-side before-sleep locking.

The Niri HM module is still imported for macOS users because the shared HM tree is imported uniformly across Linux and Darwin machines. The module therefore cannot rely on conditional imports to stay Darwin-safe. Instead it checks whether ~options.programs.niri~ exists and only materializes the ~programs.niri~ subtree when Home Manager's upstream Niri module is actually present. This avoids nix-darwin evaluation failures from defining an option path that does not exist on Darwin.

~homelab.desktop.shell~ (enum, default ~"noctalia"~) selects a desktop shell that replaces individual bar/notification/lock/wallpaper programs. When set to anything other than ~"none"~, waybar, swaync, swaylock, swaybg, and udiskie are skipped. ~swayidle~ remains available for session-side before-sleep locking, ~rofi~ remains enabled, and ~Mod+D~ continues to launch it regardless of shell selection. Compositor-level utilities (wl-clipboard, xwayland-satellite), ~swayidle~, and playerctld remain unconditional.

When ~desktop.shell = "noctalia"~, the shared Niri keymap routes shell-owned hardware controls through ~noctalia msg ...~ (the Noctalia v5 IPC entrypoint) instead of direct helper binaries: lock, volume, mute, media play/pause/stop/previous/next, brightness, Wi-Fi, Bluetooth, settings, and power-profile cycling all go through Noctalia so the shell owns the resulting state and OSD. Outside Noctalia, the same Niri config falls back to ~swaylock~, ~wpctl~, ~playerctl~, and ~light~ as appropriate. ~XF86Favorites~ mirrors ~Mod+D~ globally and still launches ~rofi -show drun~ regardless of shell.

~spawn-at-startup~ launches session processes alongside the compositor. Noctalia is first so the shell owns OSD/notifications before apps appear. Desktop apps follow, each gated on its own enable flag: ~bitwarden~ on ~role == "desktop"~, ~element-desktop~ on ~homelab.programs.element-desktop.enable~, and ~emacs --daemon~ on ~homelab.programs.emacs.enable~. The Emacs daemon pairs with a ~Mod+E~ keybinding that opens a graphical client frame via ~emacsclient -c~; the keybinding is also gated on ~homelab.programs.emacs.enable~. This keeps shell startup tied to the compositor session, which matches Noctalia's current Niri guidance and avoids enabling the shell's experimental systemd user service in parallel. A separate ~noctalia-before-sleep-lock~ user service runs ~swayidle -w before-sleep 'noctalia msg session lock'~ so lid-close suspend locks while the user session is still alive; earlier root-side sleep hooks and generic ~loginctl lock-session~ wiring were intentionally removed because they ran too late or did not integrate reliably with the active Noctalia session.

The shared Niri config defines the external LG 4K monitor using a stable logical key and matches the actual hardware by monitor name (~LG Electronics LG ULTRAFINE 110NTHM20241~) rather than a transient connector like ~DP-2~. Laptop panel placement remains machine-local in ~machines/sonic/_niri.nix~ so the shared desktop module only owns output state that is common across laptops.

The primary directional bindings use edge-aware fallbacks so movement stays continuous across the desktop topology. ~Mod+Left/Right~ and ~Mod+H/L~ focus the adjacent column, then fall through to the monitor on that side when already at the edge. ~Mod+Up/Down~ and ~Mod+K/J~ focus the adjacent window, then fall through to the workspace above or below when already at the first or last window. The matching ~Mod+Shift+...~ bindings move the focused column or window with the same fallback behavior, while the ~Mod+Ctrl+...~ bindings remain explicit monitor navigation.

* Noctalia Shell

[[https://noctalia.dev/][Noctalia]] is a minimal Wayland desktop shell. It provides panels, notifications, and system UI with a clean aesthetic. Noctalia supports Niri natively, making it a natural fit for the current compositor stack. Enabled by default (~homelab.desktop.shell = "noctalia"~). This repo tracks Noctalia *v5* (~github:noctalia-dev/noctalia~, the ~cachix~ branch), which is a ground-up rewrite of the Quickshell-based v4 line — a fresh install, not an upgrade. The system package is installed via ~modules/nixos/desktop/_noctalia.nix~; the HM module (namespace ~programs.noctalia~) is included via ~mkNixOSDesktopConfig~'s ~home-manager.sharedModules~ and activated in ~modules/home-manager/programs/niri/_noctalia.nix~; and Niri starts the shell via ~spawn-at-startup~ (~noctalia~) in ~modules/home-manager/programs/niri/default.nix~. The repo intentionally does not enable Noctalia's systemd user service so the shell has a single startup path. The flake input pins no ~nixpkgs~ ~follows~ (it carries its own nixpkgs) so builds resolve against the ~noctalia.cachix.org~ binary cache instead of compiling locally. Configuration is declared as a Nix attrset in ~programs.noctalia.settings~, serialized to ~~/.config/noctalia/config.toml~ and validated at build time (~validateConfig~, on by default); unknown keys only warn, so per-widget cosmetics can be re-captured from the GUI via ~noctalia config export~. Theming: Stylix 26.05 has no v5 target, so the Stylix base16 palette is bridged into a Noctalia v5 custom palette (~programs.noctalia.customPalettes~, mapping mirrored from Stylix's v4 noctalia-shell target) and selected via ~[theme] source = "custom"~, keeping the shell in lockstep with the desktop-wide Stylix colors. The wallpaper is not a config.toml key in v5 (it is runtime state set via ~noctalia msg wallpaper-set~), so a ~noctalia-wallpaper~ user service re-applies the Stylix image on each login. The bar layout puts workspaces/active-window/media on the left and tray/battery/volume/brightness/clock/notifications/control-center on the right, with the Bluetooth widget only rendered where ~homelab.hardware.bluetooth.enable = true~; the Control Center button shows the bundled NixOS logo. Settings are declared as a Nix attrset and captured from the running shell via ~noctalia config export~ (Noctalia layers UI tweaks over them in ~~/.local/state/noctalia/settings.toml~, so keep changes in Nix to stay reproducible). v5 removes the ~plugins.json~ registry, so the former ~model-usage~ AI-usage widget (driven by ~homelab.programs.ai_tools.models~) is not currently wired in — a follow-up pending the v5 declarative-plugin story.

* Input

Keyboard repeat is tuned to match macOS so the typing feel is consistent across the laptop fleet (NixOS desktops + Darwin machines). macOS is the source of truth — Darwin's defaults app uses 15ms ticks for ~InitialKeyRepeat~ and ~KeyRepeat~, and the equivalent libinput values are mirrored on Linux desktops via Niri's ~input.keyboard.repeat-delay~ / ~repeat-rate~.

| Setting       | Darwin (modules/macos/base.nix) | Linux (niri input.keyboard) | Effective value         |
|---------------+---------------------------------+-----------------------------+-------------------------|
| Initial delay | ~InitialKeyRepeat = 15~ (ticks)   | ~repeat-delay = 225~ (ms)     | 225 ms                  |
| Repeat rate   | ~KeyRepeat = 2~ (ticks)           | ~repeat-rate = 33~ (Hz)       | ~30 ms / repeat (~33Hz) |

Conversion: Darwin ticks × 15ms = Linux ms; ~1000 / (KeyRepeat * 15)~ ≈ Linux Hz. If you change one side, change the other — both files carry inline comments pointing at each other.

Niri's defaults (600ms delay, 25Hz) feel sluggish next to macOS, which is why we override here rather than relying on upstream defaults. Greetd's TTY prompt is unaffected, but the desktop session begins immediately after login so it does not matter in practice.

* Graphics

GPU acceleration is configured per-driver via ~homelab.hardware.graphics.driver~:

** Intel (sonic)

| Package               | Purpose                                 |
|-----------------------+-----------------------------------------|
| intel-media-driver    | iHD VA-API (hardware video, Broadwell+) |
| intel-vaapi-driver    | i965 VA-API fallback (older Intel)      |
| intel-compute-runtime | OpenCL (NEO)                            |
| libva-vdpau-driver    | VDPAU compatibility over VA-API         |
| libvdpau-va-gl        | VDPAU fallback via OpenGL               |

~LIBVA_DRIVER_NAME=iHD~. Kernel param ~i915.enable_psr=0~ disables Panel Self Refresh to fix scrolling micro-stutter on Whiskey Lake.

** AMD (usul)

| Package        | Purpose                   |
|----------------+---------------------------|
| libvdpau-va-gl | VDPAU fallback via OpenGL |

~LIBVA_DRIVER_NAME=radeonsi~. Mesa's built-in RADV/radeonsi handles VA-API natively — no separate VA-API or OpenCL driver is needed for desktop use. ~hardware.amdgpu.initrd.enable = true~ for early module loading. Kernel param ~amd_pstate=active~ forces the modern AMD P-State EPP driver for fine-grained frequency scaling.

* Audio

PipeWire replaces PulseAudio and ALSA with a unified daemon. Enabled when ~homelab.hardware.audio.enable = true~:

- ALSA compatibility with 32-bit support (Wine/Steam)
- PulseAudio compatibility layer
- RealtimeKit (~security.rtkit~) for realtime scheduling priority

* Theming

Stylix provides Base16 theming from a single source of truth at the system level (~modules/nixos/_stylix.nix~ on NixOS, ~modules/macos/_stylix.nix~ on Darwin). All settings are defined once under ~homelab.stylix.*~ and propagate to Home-Manager automatically via Stylix's ~autoImport~ (injects ~homeModules.stylix~ for every HM user) and ~followSystem~ (copies values with ~mkDefault~ priority).

Stylix is enabled by default for all machines. Override ~homelab.stylix.enable = false~ only when a host needs to opt out.

| Setting    | Default                        | Notes                                |
|------------+--------------------------------+--------------------------------------|
| Scheme     | gruvbox-dark                   | String (named) or path (custom YAML) |
| Polarity   | dark                           | ~dark~ or ~light~                        |
| Monospace  | JetBrainsMono Nerd Font (12pt) | Nerd Font variant                    |
| Sans-serif | Inter (12pt)                   |                                      |
| Cursor     | Bibata Modern Classic          | Size 24 (NixOS only, not on Darwin)  |
| Opacity    | 0.90                           | Terminal and popups                  |

NixOS system-level targets (Plymouth, GRUB, console) auto-enable when their underlying program is active. HM-only target overrides (starship disabled, waybar ~addCss = false~, zen-browser profile) are injected via ~home-manager.sharedModules~ in the system-level module.

sonic uses a custom scheme (~nix/themes/sonic-dark.yaml~) with a Sonic the Hedgehog-inspired palette.

Starship prompt is excluded from Stylix targets — it uses hardcoded Catppuccin colors. Fish uses tide instead of Starship; tide prompt colors are derived from Stylix base16 slots in ~_fish.nix~ so they adapt automatically per scheme. The prompt layout (style, separators, icons, transient prompt) is set by a home-manager activation script that runs ~tide configure --auto~ after ~writeBoundary~. The ~--auto~ flags are version-coupled to the tide release in nixpkgs (currently v6.2.0) — each flag maps to a choice file under ~functions/tide/configure/choices/~ in the tide repo, and a nixpkgs bump that adds or renames a choice will cause the activation to fail visibly.

A fontconfig override in ~home.nix~ redirects common monospace fonts (Liberation Mono, DejaVu Sans Mono, Noto Sans Mono) to the Stylix monospace font to preserve Nerd Font PUA icon glyphs in web pages and apps.

* Persistence

All desktops use impermanence with ~/nix/persist~ as the persistence root. Both current desktops (usul, sonic) use tmpfs root with LUKS-encrypted ext4, which is inherently wiped on reboot. (The shared modules still support ZFS-root desktops via ~zfs rollback -r rpool/root/empty@start~, but no current machine uses that path.) Desktop-specific persistent directories:

| Path                                   | Purpose                                                        |
|----------------------------------------+----------------------------------------------------------------|
| /var/lib/bluetooth                     | Bluetooth pairings                                             |
| /var/lib/fprint                        | Fingerprint templates (used on fingerprint-enabled desktops)   |
| /var/lib/NetworkManager                | WiFi connections                                               |
| /etc/NetworkManager/system-connections | WiFi connection configs                                        |
| /var/cache/tuigreet                    | Greeter session cache                                          |
| /home/<user>                           | User home (non-ZFS: auto-created with ownership; ZFS: dataset) |

* Kernel & Performance

Desktops use the Zen kernel (~linuxPackages_zen~) for improved interactive responsiveness. Key differences from the stock kernel:

- BORE scheduler — lower latency for foreground applications
- 1000Hz tick rate — smoother input handling and animations
- Optimized preemption — less stalling under load

ZFS 2.3.x is compatible with the Zen kernel. If issues arise, select an older generation from GRUB to revert.

** VM Tuning (sysctl)

Desktop-specific sysctl overrides inspired by CachyOS defaults. Absolute byte dirty limits replace ratio-based settings for predictable write-back regardless of RAM size. zram-aware values suit compressed swap.

| Parameter                   |     Value | Default | Rationale                                           |
|-----------------------------+-----------+---------+-----------------------------------------------------|
| vm.swappiness               |       150 |      60 | Prefer compressing to zram over evicting file cache |
| vm.page-cluster             |         0 |       3 | Single-page swap reads suit zram (no seek penalty)  |
| vm.dirty_background_bytes   |  67108864 |       0 | Flush at 64 MB background (predictable write-back)  |
| vm.dirty_bytes              | 268435456 |       0 | Hard cap at 256 MB (predictable write-back)         |
| vm.vfs_cache_pressure       |        50 |     100 | Retain more filesystem metadata in cache            |
| vm.compaction_proactiveness |         0 |      20 | Avoid background compaction latency spikes          |
| vm.min_free_kbytes          |     67584 |    4096 | Larger free-memory reserve to prevent alloc stalls  |
| kernel.nmi_watchdog         |         0 |       1 | Saves power, reduces interrupts                     |
| kernel.split_lock_mitigate  |         0 |       1 | Avoids split-lock detection performance penalty     |

** Process Scheduling (ananicy-cpp)

ananicy-cpp is an auto-nice daemon that adjusts process niceness, I/O priority, and CPU scheduling class in real time. It uses the CachyOS community ruleset (~ananicy-rules-cachyos~) which covers hundreds of common applications. A custom rule is added for ~zen-beta~ (mapped to ~Doc-View~ type, same as Firefox/Zen) since upstream only covers ~zen~ and ~zen-bin~.

** Gaming

Gated on ~homelab.desktop.gaming.enable~. The module lives in ~modules/nixos/desktop/_gaming.nix~ and is NixOS-level (not Home Manager) because Steam, GameMode, and Gamescope require system-wide capabilities, udev rules, firewall ports, and a 32-bit driver stack.

*** What it enables

| Component                 | NixOS option / package              | Purpose                                                   |
|---------------------------+-------------------------------------+-----------------------------------------------------------|
| Steam                     | ~programs.steam~                      | Game store, Proton runtime, Steam Input                   |
| Proton-GE                 | ~extraCompatPackages = [proton-ge-bin]~ | Community Proton build with extra game fixes, media codecs |
| Protontricks              | ~programs.steam.protontricks~         | Winetricks wrapper for per-prefix tuning                  |
| extest                    | ~programs.steam.extest~               | X11 → uinput translation for Steam Input on Wayland       |
| Gamescope                 | ~programs.gamescope~                  | Per-game compositor for resolution scaling, frame limiting |
| GameMode                  | ~programs.gamemode~                   | On-demand CPU/IO/GPU optimizations when games launch      |
| Steam hardware udev rules | ~hardware.steam-hardware~             | Controller support (Steam Controller, etc.)               |
| 32-bit graphics drivers   | ~hardware.graphics.enable32Bit~       | Mesa, Vulkan loaders, VA-API for Proton/Wine              |
| MangoHud                  | ~environment.systemPackages~          | Vulkan overlay for FPS/GPU/CPU stats                      |
| ProtonUp-Qt               | ~environment.systemPackages~          | GUI for managing additional Proton-GE versions            |

Firewall ports are opened for Steam Remote Play and Steam Local Network Game Transfers.

*** GameMode settings

GameMode renices game processes by -10 and sets I/O priority to BEST_EFFORT/0. GPU-specific tuning is conditional on ~homelab.hardware.graphics.driver~:

- *AMD* (~usul~): sets ~power_dpm_force_performance_level = "high"~ via sysfs for the duration of the game session.
- *Intel* (~sonic~): GPU tuning is skipped. Intel iGPUs (UHD 620) lack the DPM sysfs interface that GameMode targets. CPU and I/O optimizations still apply.

Both ~gamescope~ (via ~capSysNice~) and ~gamemoded~ (via ~enableRenice~) receive ~CAP_SYS_NICE~. These are independent capabilities on separate binaries and do not conflict.

*** Persistence

Steam data lives entirely under ~$HOME~ (~/.local/share/Steam/~, ~/.steam/~). Since ~/home/<user>~ is already persisted on both desktops via impermanence, no additional bind-mount configuration is needed. Games, Proton prefixes, save files, and shader caches all survive reboots.

*** Niri window rules

Niri window rules in ~modules/home-manager/programs/niri/default.nix~ handle Steam and game windows:

- *Steam client* (~steam~, ~Steam~) — ~open-floating = true~. The store/library UI floats as a regular desktop window without tiling.
- *Proton/Wine games* (~steam_app_*~) and *Gamescope* — ~open-fullscreen = true~. Games open fullscreen via Niri so XWayland cursor coordinates map correctly. Without this, games open as tiled or floating windows with misaligned mouse input.

The ~steam_app_~ match is an anchored prefix (no ~$~) because Proton games report their app-id as ~steam_app_<appid>~ (e.g. ~steam_app_813780~).

*** Usage

- *MangoHud*: launch a game with ~MANGOHUD=1 %command%~ in Steam launch options, or enable globally via ~MANGOHUD=1~ in the environment.
- *GameMode*: launch a game with ~gamemoderun %command%~ in Steam launch options. Steam games using the GameMode API activate it automatically.
- *Gamescope*: launch a game with ~gamescope -W 1920 -H 1080 -- %command%~ for resolution scaling, or use Steam's per-game Gamescope integration.

* Boot

ZFS desktops use GRUB (~gfxmodeEfi = "auto"~ for graphical framebuffer); Stylix themes the GRUB menu with Base16 colors by default. Non-ZFS desktops (usul, sonic) use systemd-boot with systemd-based initrd and Plymouth. The boot sequence there is: systemd-boot → Plymouth splash (themed by Stylix) → LUKS passphrase (via Plymouth's password agent) → desktop session. If the machine explicitly disables Stylix, the boot path remains the same but the Base16 theming pieces fall away. If the machine is an unencrypted or ZFS desktop, greetd falls back to the normal greeter path instead. Console output is suppressed by ~boot.consoleLogLevel = 3~ (warnings+ only) and kernel params ~quiet~, ~rd.udev.log_level=3~, ~systemd.show_status=auto~. Note: the log level must be set via ~consoleLogLevel~, not a ~loglevel=N~ kernel param, because the NixOS kernel module unconditionally appends ~loglevel=<consoleLogLevel>~ after all ~kernelParams~ entries and the kernel honours the last value. Plymouth's NixOS module adds ~splash~ automatically. Plymouth and systemd initrd are auto-enabled for any machine with ~role = "desktop"~ and ~isRootZFS = false~ (the ~isDesktopSystemdBoot~ guard in ~_boot.nix~).

* Pain Points

** Fingerprint login cannot unlock GNOME Keyring

When ~pam_fprintd.so~ succeeds (marked ~sufficient~), PAM short-circuits the auth stack. ~pam_gnome_keyring.so~ never receives a password, so the ~login~ keyring stays locked. Apps that use the Secret Service API (Element, Discord) will prompt for credentials each session.

Note that the analogous auto-login pain point on encrypted non-ZFS desktops (~usul~, ~sonic~) is now resolved by ~pam_fde_boot_pw~ in the greetd session stack — see Login Flow / GNOME Keyring above. The same trick does not help fingerprint logins because the only password source available to PAM during a fingerprint login is the user's typed password, which is exactly what fingerprint replaces. So on ~sonic~ (encrypted non-ZFS desktop with a fingerprint reader), password logins unlock the keyring automatically via ~pam_fde_boot_pw~, but fingerprint logins still leave the keyring locked.

*Workaround:* log in with password instead of fingerprint. The keyring unlocks automatically.

** Bitwarden system-auth unlock is no longer tied to fingerprint

Previously ~com.bitwarden.Bitwarden.unlock~ was installed only when ~homelab.hardware.fingerprint.enable = true~, which coupled Bitwarden's system-auth path to fingerprint hardware and left non-fingerprint desktops (~usul~) without it. This is now controlled by the dedicated ~homelab.programs.bitwarden.systemAuthUnlock~ option (see Bitwarden Desktop above), independent of fingerprint support. The ~passwordless~ sub-option additionally makes the unlock silent — see the security tradeoff noted there.

** Bitwarden system-auth auto-unlock does not work end-to-end

Goal: have Bitwarden Desktop auto-unlock silently after boot on the autologin desktops (~usul~, ~sonic~), so the vault is ready without typing the master password. The ~systemAuthUnlock~ option (~enable~ + ~passwordless~) was built for this and is deployed, but as of 2026-07 the vault still prompts for the master password after every restart. The reason is upstream, not our config.

What we verified is *correct* on ~usul~ (all confirmed live):
- Polkit action ~com.bitwarden.Bitwarden.unlock~ installed with ~allow_active = yes~ (silent grant).
- A polkit authentication agent is running in the Niri session (~polkit-kde-authentication-agent-1~).
- The login keyring is unlocked at boot (~pam_fde_boot_pw~); ~desktop_core~ logs ~OS keyring is_locked=false~.
- Bitwarden's biometric key is present in the keyring (secret service item ~service=Bitwarden_biometric~, ~account=<userId>_user_biometric~) and ~data.json~ has ~biometricUnlockEnabled=true~ with a stored ~clientKeyHalf~.

Despite all that, the lock screen's "Unlock with system authentication" button is *disabled*. Two independent upstream walls:

1. *Feature flag.* Desktop ~2026.5.0~ carries a ~LinuxBiometricsV2~ flag (~pm-26340-linux-biometrics-v2~), default ~false~, only turned on if the server advertises it in ~/api/config~ ~featureStates~. Our server is Vaultwarden ~1.36.0~ (the current latest), whose ~SUPPORTED_FEATURE_FLAGS~ whitelist includes ~pm-25373-windows-biometrics-v2~ but *not* the Linux one — so we cannot even set it via ~EXPERIMENTAL_CLIENT_FEATURE_FLAGS~ without Vaultwarden refusing to start.
2. *Client status logic.* Forcing the flag ~true~ locally (patching the client's cached ~serverConfig.featureStates~ in ~data.json~) did *not* enable the button, so on this desktop build the flag is necessary-but-not-sufficient — the renderer's biometric-status check refuses Linux system-auth for another reason we did not pin down.

Because both walls are upstream (Bitwarden desktop + Vaultwarden), the option is left in place dormant rather than reverted: the NixOS/polkit groundwork is already correct, so if a future Vaultwarden whitelists the flag *and* a future desktop build enables the button, silent auto-unlock should start working with no further Nix changes. Re-test by launching Bitwarden and checking whether that button is enabled.

** ZFS requires GRUB

systemd-boot does not support ZFS. Both current desktops (usul, sonic) use ext4+LUKS and systemd-boot; the ZFS-root desktop path (GRUB) is still wired in ~modules/nixos/_boot.nix~ but has no live machine today.
