#+TITLE: Secrets Rotation & Incident Response
#+STARTUP: overview

* Architecture Overview

Secrets are managed with [[https://github.com/Mic92/sops-nix][sops-nix]] using age encryption.

** Key Hierarchy

| Key             | Location                            | Encrypts                        |
|-----------------+-------------------------------------+---------------------------------|
| masoodahmed     | =~/.config/sops/age/keys.txt= (macOS) | All secrets (admin key)         |
| murderbot       | =~/.config/sops/age/keys.txt= (macOS) | All secrets (admin key)         |
| Per-machine key | =/nix/secret/age/ssh_ed25519_key=     | That machine's secrets + shared |

** Secret Tiers

| Tier     | File                                  | Encrypted For             |
|----------+---------------------------------------+---------------------------|
| Global   | =secrets/secrets.sops.yaml=             | All NixOS machines + devs |
| Scoped   | =secrets/headscale-{dns,acl}.sops.yaml= | meshcontrol, trialunit    |
| Per-host | =machines/<name>/secrets.sops.yaml=     | That machine + devs       |

* Routine Rotation

** Rotate All Encrypted Values (quarterly)

Re-encrypts every sops file with current keys. Does not change the secrets themselves, but ensures key material is fresh.

#+begin_src bash
just sops-rotate
#+end_src

This runs =sops --rotate --in-place= on every =*.sops.yaml= file.

** Rotate a Specific Secret

1. Edit the sops file:
   #+begin_src bash
   sops machines/heartbeat/secrets.sops.yaml
   #+end_src
2. Change the secret value.
3. Commit and deploy:
   #+begin_src bash
   git add machines/heartbeat/secrets.sops.yaml
   git commit -m "chore(secrets): rotate heartbeat/<secret-name>"
   just deploy machine=heartbeat
   #+end_src

** Rotate Application Credentials

Some secrets are credentials for external services. When rotating these, update both the sops file and the external service:

| Secret                | External Service             | Notes                          |
|-----------------------+------------------------------+--------------------------------|
| =cloudflare/api-key=  | Cloudflare dashboard         | ACME DNS challenge             |
| =grafana/authentik-*=   | Authentik admin -> Providers | Regenerate client secret       |
| =headscale/oidc-*=      | Authentik admin -> Providers | Regenerate client secret       |
| =vaultwarden/.env=      | Authentik admin -> Providers | SSO client secret inside       |
| =authentik/.env=        | Authentik internal           | Secret key, email creds        |
| =restic/*=              | Garage S3 credentials        | Access key in Garage admin     |
| =tailscale/preauth-key= | Tailscale admin console      | Generate new key, revoke old   |
| =mongodb/root-password= | Internal (localhost only)    | Restart mongodb after rotating |

* Adding a New Machine

1. Generate an age key on the new machine:
   #+begin_src bash
   nix-shell -p age --run "age-keygen -o /nix/secret/age/ssh_ed25519_key"
   #+end_src

2. Copy the public key (=age1...=) from the output.

3. Add the key to =.sops.yaml=:
   - Add an anchor under =keys:=
   - Add it to relevant =creation_rules= (at minimum the machine's own regex + global)

4. Re-encrypt affected files so the new machine can decrypt:
   #+begin_src bash
   just sops-update
   #+end_src

5. Commit =.sops.yaml= and all re-encrypted =*.sops.yaml= files.

* Decommissioning a Machine

1. Remove the machine's age key anchor from =.sops.yaml=.
2. Remove it from all =creation_rules= entries.
3. Re-encrypt to revoke access:
   #+begin_src bash
   just sops-update
   just sops-rotate
   #+end_src
4. Rotate any secrets that were shared with the decommissioned machine (at minimum: =secrets/secrets.sops.yaml= values).
5. Commit everything.

* Incident Response: Compromised Machine Key

If a machine's age key is compromised, that machine could decrypt any sops file it was a recipient of. The blast radius depends on the tier:

** Step 1 -- Assess Blast Radius

Check =.sops.yaml= to determine which =creation_rules= include the compromised key.

- *Per-host secrets only*: only that machine's =machines/<name>/secrets.sops.yaml=
- *Global secrets*: =secrets/secrets.sops.yaml= (all shared secrets exposed)
- *Scoped secrets*: e.g., headscale secrets if the machine was meshcontrol/trialunit

** Step 2 -- Revoke the Key

1. Remove the compromised key from =.sops.yaml=.
2. Re-encrypt all affected files:
   #+begin_src bash
   just sops-update
   just sops-rotate
   #+end_src

** Step 3 -- Rotate Exposed Secrets

Every secret the compromised key could decrypt must be assumed leaked. Rotate:

1. All values in =secrets/secrets.sops.yaml= (if global recipient).
2. All values in the machine's =machines/<name>/secrets.sops.yaml=.
3. All values in any scoped files (=headscale-dns=, =headscale-acl=) if applicable.
4. External credentials (Cloudflare API key, Tailscale preauth key, OAuth client secrets, restic repo credentials, etc.).

** Step 4 -- Re-key the Machine

1. Generate a new age key on the machine.
2. Follow the "Adding a New Machine" procedure above.
3. Deploy to the machine with the new key.

** Step 5 -- Verify

1. Confirm the machine can decrypt its secrets (deploy succeeds).
2. Confirm all services start correctly.
3. Confirm the old key no longer appears in =.sops.yaml=.

* Incident Response: Compromised Admin Key (masoodahmed/murderbot)

This is the worst case -- admin keys can decrypt *all* secrets.

1. Generate a new age key on the compromised dev machine.
2. Update =.sops.yaml= with the new public key.
3. =just sops-update= to re-encrypt for the new key.
4. *Rotate every secret in every sops file.* All values must be treated as leaked.
5. Rotate all external credentials (Cloudflare, Tailscale, OAuth, S3, etc.).
6. Deploy to all machines.

* macOS Key Location

On macOS (murderbot, work-pantheon), the age key lives at:

#+begin_src
~/.config/sops/age/keys.txt
#+end_src

Back this up securely. If lost, you must re-key the machine (generate new key, update =.sops.yaml=, re-encrypt).
